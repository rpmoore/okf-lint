# Code Review: Section 01 - Foundation

Overall the implementation is a faithful, mostly correct match to the plan: Cargo.toml deps (clap/derive, walkdir, serde_yaml_ng, chrono, plus dev-deps assert_cmd/predicates/insta) all present at the specified major versions, diagnostic.rs's Rule/Diagnostic types and both required tests exist, frontmatter.rs's split_frontmatter and all five required tests exist and are logically correct (traced body_start_line math by hand for the Found/Unclosed/blank-line/trailing-char cases — all check out), the LintError stub correctly lives in src/lint.rs (not walk.rs) as mandated, and walk.rs's collect_md_files + its five tests match the spec (dot-file/dir exclusion via filter_entry, .md-only filtering, root-relative paths, lexicographic sort, permission-denied -> LintError::Io). main.rs wiring (mod diagnostic/frontmatter/lint/walk) is present.

Criticisms a senior architect would raise:

1. CLAUDE.md violation (project-mandated, not optional): the project's own CLAUDE.md requires updating `docs/knowledge/` for the section of code touched after every code change, creating a doc if none exists. This diff touches diagnostic.rs, frontmatter.rs, lint.rs, and walk.rs but adds zero files under docs/knowledge/ (confirmed the directory doesn't exist at all in the repo). This is a straightforward, explicit process requirement that was simply skipped.

2. `split_frontmatter` (src/frontmatter.rs:102-123) only splits on `'\n'` and does exact `line == "---"` matching. Any file with CRLF line endings will have the delimiter line read as `"---\r"`, which fails the exact match and silently downgrades a well-formed frontmatter block to `FrontmatterResult::None` (or a spurious `Unclosed`) instead of `Found`. Neither the plan nor the tests exercise CRLF, but this is a realistic real-world input (Windows-authored Markdown) that will silently misclassify valid frontmatter with no error/warning — worth flagging even though it wasn't explicitly required in the plan.

3. `permission_denied_subdirectory_is_io_error` (src/walk.rs:328-347) will not actually exercise the failure path when the test suite runs as root (common in CI containers/Docker), since root bypasses Unix permission bits — `chmod 0o000` has no effect and the walk will succeed, silently passing on an assertion that never got a chance to fail meaningfully, or worse, matching `Err` for an unrelated reason. No `#[cfg]`/runtime check guards against running as root, making this test environment-fragile with no diagnostic when it's not actually testing what it claims to.

4. Test cleanup in walk.rs's temp-dir helper tests uses `fs::remove_dir_all` at the end of each test body rather than an RAII guard; if an assertion earlier in the test panics, cleanup is skipped and stray `okf-lint-walk-test-*` directories accumulate in the OS temp dir across CI runs. Low severity but a known anti-pattern the plan's own suggested alternative (`tempfile` crate, explicitly permitted) would have avoided.

5. Minor deviation: `src/lint.rs:182` blanket-applies `#[allow(dead_code)]` to the whole `LintError` enum, silencing the compiler warnings the plan explicitly says to 'expect and ignore' (i.e., leave visible) for the three yet-unused variants. Not wrong, but suppressing rather than surfacing them removes a signal that would otherwise remind implementers those variants are still unwired until section-06.

No structural or API-shape deviations from the plan were found (Rule ordering, Diagnostic fields, FrontmatterResult variants, collect_md_files signature, and the LintError stub all match precisely), and no data-loss/crash-class bugs were found beyond the CRLF handling gap noted above.
